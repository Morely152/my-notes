# 二、编写Skill

新建文件`SKILL.md`，务必保证全大写，以免Agent无法识别到；然后写入Skill的内容。可以让AI来编写Skill，但是**务必保证元数据中包含 name 和 description 字段**，然后描述Skill的功能和步骤，最后可以加上一个具体的例子。

```text
@SKILL.md 帮我写完这个新闻总结的SKILL，提供新闻总结的技巧和要点，提醒Agent务必保证内容的准确性、严谨性和时效性，然后整理成一张表格，只需要生成表格即可，不需要总结文字。最后给出一个例子来具体说明。让Agent将表格保存为当前目录下的news.md中，以这个文件作为交付给用户的成品。
```

name: 新闻总结助手

# 新闻总结助手

## 技能概述

本技能专注于将新闻文章转化为结构化的Markdown表格，帮助用户快速把握新闻要点。仅输出表格格式，不包含任何文字总结。

## 核心原则 ⚠️

**在执行新闻总结任务时，必须严格遵守以下原则：**

### 1. 准确性（Accuracy）
- **忠实原文**：所有信息必须直接来源于新闻文本，不得添加原文中没有的内容
- **准确引用**：引用具体数据、时间、数字时，必须与原文完全一致
- **避免推测**：不得对新闻内容进行主观推测或延伸解读
- **验证信息**：对于存疑的信息，应在表格中标注"待确认"或"据报道称"

### 2. 严谨性（Rigor）
- **客观中立**：保持客观立场，不加入个人观点或情感色彩
- **逻辑清晰**：确保事件、地点、人物之间的逻辑关系准确
- **区分事实与观点**：明确区分客观事实报道与主观评论
- **避免歧义**：使用清晰、准确的语言，避免模糊表述

### 3. 时效性（Timeliness）
- **注明时间**：所有事件必须标注明确的时间点（日期、时间）
- **时间顺序**：按照时间先后顺序排列事件
- **时效意识**：注意新闻的发布时间与事件发生时间的关系
- **进展追踪**：对于持续事件，注明"截至报道时"的状态

## 新闻总结技巧

### 信息提取要点

#### 📌 识别关键要素
- **时间**：精确到具体日期和时刻
- **地点**：具体的城市、区域、场所
- **人物**：完整姓名、职务、身份
- **事件**：简洁概括事件核心内容（50字以内）
- **原因**：事件发生的背景或导火索
- **结果**：事件的直接后果或影响
- **数据**：相关统计数字、百分比、数量

#### 📌 内容筛选
- **优先保留**：突发事件、重大影响、涉及公共利益的事件
- **适当精简**：背景介绍、细节描述可适当概括
- **合并同类项**：多个相似事件可合并为一条记录
- **标注关联**：相关事件应在备注中说明关联关系

#### 📌 语言处理
- **保持原意**：不改变原文的核心含义
- **简洁表达**：用最少的文字传达完整信息
- **专业术语**：保留专业术语，必要时可加注解释
- **数字格式**：统一使用阿拉伯数字

## 输出格式规范

### 表格结构

使用以下标准Markdown表格格式：

| 时间 | 地点 | 人物/机构 | 事件概述 | 影响/结果 | 备注 |
|------|------|----------|---------|----------|------|
| ... | ... | ... | ... | ... | ... |

### 字段说明

| 字段 | 说明 | 要求 |
|------|------|------|
| 时间 | 事件发生的时间 | 精确到年月日，如有具体时间请标注 |
| 地点 | 事件发生的地点 | 具体到国家、省/州、城市、场所 |
| 人物/机构 | 主要参与方 | 列出关键人物姓名及职务，或机构名称 |
| 事件概述 | 事件核心内容 | 50字以内简洁概括，突出核心事实 |
| 影响/结果 | 事件的直接后果 | 量化数据优先，如无数据可简述影响 |
| 备注 | 补充说明 | 信息来源、事件进展、相关链接等 |

## 处理特殊情况

### 多条事件处理
1. **独立事件**：每条事件单独成行
2. **系列事件**：按时间顺序排列，在备注中注明关联
3. **持续性事件**：标注起止时间，区分不同阶段
4. **未确认信息**：在备注中标注"据媒体报道""待官方确认"等

### 信息缺失处理
- 时间不明确：标注"时间待确认"
- 地点模糊：标注"某地"或原文表述
- 人物未知：标注"相关人员"或"不明身份人士"
- 数据冲突：列出多个来源数据并注明出处

## 质量检查清单

在提交总结前，请检查：
- [ ] 所有信息均有原文依据
- [ ] 时间、地点、人物准确无误
- [ ] 数字、数据与原文一致
- [ ] 客观中立，无主观评论
- [ ] 语言简洁，无冗余信息
- [ ] 表格格式规范完整
- [ ] 备注信息充分必要

# 七、渐进式披露

> **How Skills work:** When Claude encounters a task, it scans available Skills to find relevant matches. Skills use progressive disclosure: metadata loads first (~100 tokens), providing just enough information for Claude to know when a Skill is relevant. Full instructions load when needed (<5k tokens), and bundled files or scripts load only as required. （**技能的工作原理：** 当 Claude 遇到一个任务时，它会扫描可用的技能以找到相关匹配。技能使用渐进式披露：元数据首先加载（约 100 个 token），提供足够的信息让 Claude 知道何时技能是相关的。完整指令在需要时加载（约 3.5k 个 token），捆绑的文件或脚本仅在需要时加载。）[^3]

通过给Agent编写这样一个将新闻文章总结成表格，并且按需加入审核和图片转换的功能，我们意识到Agent Skills的本质是**对prompt的精细化管理**，每一次处理都让Agent“精准选择”需要的信息给LLM，任务之外的prompt则不会进行传输，从而节省了token的开支。也就是说，Agent Skill的设计是一个“渐进式披露（Progressive Disclosure）”的结构：

- 元数据层：集合了所有Skill的名称和数据，Agent始终加载，相当于是一个总体的目录
- 指令层：对应每个SKILL.md中的内容，只有LLM通过Function Calling告知LLM需要这样的Skill时，才会被Agent传输给LLM
- 资源层：包含Skill中的Reference和Script，是更细一层的控制，只有当用户的需求命中了Skill中的调用条件时，LLM才会让Agent发送对应的参考文本，或者执行对应的脚本来进行处理。

这三层就像三个上下摆放的“筛子”，时刻保证“筛选”给LLM的prompt都是完成任务所必须的信息，多余的内容都会被拦截在上一层中，Agent只会把LLM需要的东西给它。

# 八、Skill与MCP

Skill与MCP有一定的共同性，两者都是为了让Agent能够调用工具而创造的，但是在一定程度上也存在着差异：

| 项目    | MCP             | Agent Skills |
| ----- | --------------- | ------------ |
| 本质    | 传输协议（程序）        | 文档说明（文档）     |
| 功能    | 给大模型提供数据        | 说明如何处理数据     |
| 关键价值  | 标准化连接、权限/隔离、可扩展 | 渐进式披露、复用工作流  |
| 上下文信息 | 往往需要把工具定义提前暴露   | 可按需加载（漏斗式）   |
| 风险点   | 工具滥用、权限配置不当     | 提示注入、脚本乱跑    |
| 适合场景  | 稳定连接外部系统/数据源    | 重复性工作、步骤明确   |

> Skill 和 MCP 是互补关系：MCP 像“接口/USB-C”，Skill 像“带说明书的配件包”。[^2]

Mengke大佬形象地将MCP比喻为“纯粹的**USB接口**”，而Skill更像是“拥有转接头、驱动、说明书等内容的**拓展坞套装**”。Skill 的核心不是“再定义一套工具协议”，而是提供了一份**可复用的SOP**（标准作业程序，Standard Operating Procedure[^4]）[^2]。

Claude官方发布的文章中，对“何时用Skill取代MCP”这一问题做出了如下的解释：

> Use both together: MCP for connectivity, Skills for procedural knowledge.(两者结合使用：MCP 用于连接，技能用于流程知识。)[^3]

Skill 指导Agent“使用工具进行哪些操作”，而“如何使用工具”，则由MCP负责。
