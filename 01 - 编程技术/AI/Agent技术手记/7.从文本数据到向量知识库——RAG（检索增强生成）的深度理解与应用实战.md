伴随着各种大模型的兴起，许多平台开始搭建基于私有知识的AI问答智能体，如售前咨询客服、产品介绍助手、文档快随查询等。这些智能体或多或少都用到了一个共同的技术——RAG（Retrieval-augmented Generation，检索增强生成）。本篇笔记将根据我搭建个人博客AI助手模块的经验和学习成果编写，从RAG系统的定义和特点开始，历数文本向量化的实现原理、向量检索匹配的实现原理、通过静态知识库、意图识别、问题轻量改写和关键词/语义匹配加权排序检索等方案优化RAG的知识召回效果等内容。                         
# 一、缘起：为什么我们需要RAG

![](20260611224859969.png)

自2024年AI大模型进入“百家争鸣”的局面以来，国内外涌现出了一大批功能强大、特色各异的大模型，如GPT、Gemini、DeepSeek、豆包、Kimi等。人们逐渐习惯于通过与大模型对话来获取知识，各大平台也开始引入AI客服，应用到售前咨询、文档解读等领域，从而借助AI大模型的智能回答能力让用户快速、准确地获得问题解答，用户就无需像往常一样去翻阅冗长复杂的文档。于是RAG技术应运而生——让大模型“读懂”私有知识，从而增强它输出内容的质量。
## 1.通用大模型如何“读懂”私有知识

“读懂”这个词似乎有点难以理解，这里举一个具体的例子来说明。想象一下我们给一家保险公司的网站接入了AI大模型，用户可以在网站上提问，问题交给大模型进行回答。

```
用户：你好！
        ↓
---- 保险公司系统后台 -------------
用户提示词：你好！
系统提示词：你是一位XX保险公司的专业
客服，请你以专业热情的态度解决用户提
出的问题。
-----------------------------------
        ↓
---- AI大模型 ---------------------
返回回答：您好，我是XX保险公司的客服
，请问您需要咨询什么内容？
-----------------------------------
```

某天用户询问了“我可以选购哪些保险服务？”，结果系统背后的大模型直接上网查询了其他公司的保险政策文件，瞎编了一大串不属于甲方公司的服务，因此我们的AI智能客服遭到了用户的投诉，甲方责令我们进行修复。

为了解决这个问题，我们提取了`《XX保险公司用户协议合同》.pdf`中的文本，将其添加到了系统提示词中，现在AI大模型就会参考协议合同中的内容进行回答：

```
用户：我可以选购哪些保险服务？
        ↓
---- 保险公司系统后台 -------------
用户提示词：我可以选购哪些保险服务？
系统提示词：你是一位XX保险公司的专业
客服，请你以专业热情的态度解决用户提
出的问题。以下是公司的用户协议合同，
需要参考其中的内容进行解答，协议合同
中没有用户问题相关的内容时，严禁参考
网络知识或者训练知识进行解答，需要进
行拒绝回答，并引导用户询问其他相关的
内容。
附件：《XX保险公司用户协议合同》.txt
-----------------------------------
        ↓
---- AI大模型 ---------------------
返回回答：根据我司用户协议合同内容，
您可以投保以下类型的保险服务：...
-----------------------------------
```

通过直接把文档**全文**添加到提示词当中，现在的AI智能客服就学会了参考我们提供的私有知识来回答，而非参考不准确的知识，从而出现“幻觉”现象。

> **大模型幻觉（LLM Hallucination）** 指大语言模型在生成文本时，产生与用户输入相悖、与已有上下文矛盾、或与真实世界知识不符内容的现象。术语来自神经科学，类比大脑在无外部刺激时自发产生感知的病理状态。[^1]

甲方很高兴，通过了验收。事实上这样就形成了RAG的雏形——给大模型提供知识库，使大模型从中检索相关内容作为参考，从而准确回答用户的问题。但这毕竟是雏形，事实上一个成熟的RAG系统比这更加复杂。

## 2.给大模型写“说明书”：构建知识库

一段时间后，甲方业务范围扩展，从原来只做人身保险增加了财产保险，下设企财险、工程险、车险、责任险、船舶险、货运险、家财险、信用险、保证险、农险、巨灾保险、新能源汽车专属保险[^2]等一大堆业务，而且各自有对应的保险政策和用户协议，现在需要把这些内容同步给AI智能客服。

看着这么几十个文件，现在把它们全部放到提示词里面，每次都发给AI显然不是一个明智的办法。一方面用户问题只涉及某些文件中的一些内容，每次都发全量文档参考会造成不必要的流量和Token支出；一方面大模型基于数学概率，让它在海量知识中检索相关内容也存在很多不确定性，不能保证每次都能找到有用的知识。

既然如此，我们选择把每篇文章都切碎成一些文本块，然后贴上标题、分类、时间等标签（元数据），并且转换为**向量数据**存入向量数据库中存储起来。当用户提问时，由系统来检索其中相关的内容，然后把这些内容发送给大模型进行参考。这样就相当于给大模型定制了一本“说明书”，大模型得以参考其中相关的知识来回答用户的问题。

![](20260611220956957.png)

## 3.查找“说明书”：使用向量检索

为什么要将文本块转换为向量？为了知道发送哪些相关的知识文本块给大模型进行参考，系统需要具备从知识库中检索相关内容的能力。需要注意这里的“相关”指的是**语义上**的相关性，而不是字面意义上的匹配。因此使用关键词匹配显然不够，需要内置一个embedding模型将文本转换为一系列特征值，组成**高维向量**，语义相似的文本对应的向量，在高维空间中的位置也越近。在RAG系统中，通常使用余弦相似度算法（后文将会详细介绍），即通过计算两个向量夹角的余弦值来衡量它们的相似性，其值越接近1表示越相似，越接近-1表示越不相似。

于是，对于切分好的文本块，我们使用embedding模型将其转换成向量，即一串表示文本特征的数字，随着文本块一起存入向量数据库中，这样就构建了更加智能的知识库——向量知识库。

![](20260611223314024.png)
## 4.RAG概述的总结

通过上述分析，我们了解到了传统的将文档直接塞进提示词的做法，会随着文档内容的不断增加而被淘汰，进而采取将海量文本切块，进行向量化处理的方法来构建向量知识库。用户提问时，系统也抽取问题的向量，与知识库中的向量进行相似度匹配，达到从语义的角度来检索相关内容的目的，而不是简单检索包含问题关键字的相似内容。了解了RAG的大致流程后，我们接下来将更加详细地分析从知识入库到向量检索整个过程的原理与实现。接下来，我们将深入了解这一过程，以及整个RAG模块是如何在我的博客系统中实现的。
# 二、构建：从文本到知识库的实现过程

## 1.embedding模型简介

我们需要实现这样的处理：将一些句子，根据它们的意思（语义）来“定位”到一个坐标系中；意思越接近，它们的位置就越靠近。这样当我们需要在知识库中查找跟用户的问题“意思接近”（语义相似）的知识时，先“定位”用户问题在坐标系中的位置，然后返回问题附近的一些句子就可以了。这样的工具称为“**embedding模型**（一般称作嵌入模型、向量化模型）”。与大语言模型LLM不同，它接收到句子之后，输出的不是句子，而是一组元素个数相同的浮点数数组。数组中的这些元素（一般有成百上千个）可以对应一个高维空间中的坐标，于是我们将这个由很多元素组成的向量称为“**高维向量**”，数组中元素的个数称为向量的维度。

对于同一个embedding模型而言，无论输入的句子长度如何，它输出的高维向量的维度是固定的，并且向量中各个维度的分量含义也是对应相同的。例如在一个embedding模型中，第1维可能专门负责探测“情绪积极程度”；第5维可能专门负责探测“文本是否涉及科技领域”。第233维可能专门负责探测“句子的复杂程度”。这样使得由同一个embedding模型转化而来的一些向量之间具有了良好的一致性。

例如，我们有一个简易的embedding模型，它会把每个输入的句子转化为二维向量，两个维度分别为 `(情绪积极程度, 科技相关程度)`，取值范围 `[-1, 1]`。根据这样的规则，对下面三句话进行处理：

|句子|输出向量|解读|
|---|---|---|
|**我喜欢吃苹果**|**(0.8, -0.3)**|情绪积极（喜欢），但"吃苹果"是生活场景，几乎不涉及科技，所以科技维度偏负|
|**我不喜欢学编程**|**(-0.6, 0.9)**|情绪消极（不喜欢），但"编程"是典型科技活动，所以科技维度很高|
|**苹果手机好用**|**(0.7, 0.8)**|情绪积极（好用），且"苹果手机"是科技产品，科技维度也高|

当用户问系统“我喜欢吃什么？”的时候，embedding模型认为这个问题情绪很积极，与科技不相关，得到`(0.5，-0.5)`的向量，于是就会寻找这个位置附近的知识，从而发现“我喜欢吃苹果”，返回了“苹果”的回答。

## 2.文本向量化原理：从文字到数字

文本向量化技术伴随embedding模型的推陈出新不断演进，衍生出了这样三个层层递进的核心范式：
### ① 第一阶段：上下文无关（Context-Free）—— 静态表示

此阶段的代表技术有One-hot（独热编码）、Bag-of-Words（词袋模型）、TF-IDF等。
#### Bag-of-Words（词袋模型）

词袋模型的核心思想可以追溯到1954年，由语言学家泽里格·哈里斯（Zellig Harris）在其著作《Distributional Structure》中提出。其核心思想是把文章拆散为无数个词语，按照它们出现的频率进行统计，得到词频表作为特征向量。从定义上来说，词袋模型将文档表示为向量 $x \in R^{d}$ （一个$d$维的向量$R$），其中维度 $d$ 是词汇表中词汇的数量，而向量中的每个分量 $x_{j}$ 的值就是文档中单词 $j$ 出现的次数。这样处理之后，一个句子或者一篇文章就被整理好装进了“袋子”中。

```text
词袋模型向量化示例
---------------------
输入文本：“天天忙，忙完今天忙明天，忙不完。”
分词操作：[天天，忙，忙，完，今天，忙，明天，忙，不，完]
建立词表：[天天, 忙, 完, 今天, 明天, 不]
统计词频：
		天天：1
		忙  ：4
		完  ：2
		今天：1
		明天：1
		不  ：1
得到结果：[1, 4, 2, 1, 1, 1]
```

词袋模型最大的缺点便是认为所有单词都具有相同的重要性，这导致词袋模型在处理自然语言的过程中出现了这些问题：

- **忽略了词汇的顺序**
	- “A比B快”与“B比A快”的词频统计相同，因此会被词袋模型认为是同样语义的内容。
- **忽略某些词汇比其他词汇更相似的事实**
	- “我有一只狗”、“我有一只猫”、“我有一只气球”，词袋模型无法分析得出前两个句子更相关。
- **无法区分词义**
	- “这是一个苹果手机”、“这是一个苹果树”在词袋模型看来是很相关的两句话。

即使词袋模型有着诸多不足，它仍然堪当文本向量化技术发展中的一块重要的里程碑。词袋模型开创性地跨越了文本的离散型与向量的连续性，让距离、夹角、矩阵运算等数学工具在文本上有了用武之地；词袋模型作为向量空间模型的最简实现，其核心思想"**文档 = 词的加权向量**"至今仍是所有文本表示方法的元模型；词袋模型将文本相似度匹配算法的复杂度从字符串匹配的 $O(n)$ 降低到了求向量点积的 $O(n)$ ，在当时算力条件不足的背景下是一次技术飞跃。

现如今词袋模型凭借速度快、内存低、无监督的极简主义思想，仍然在垃圾邮件过滤、文本情感分析、NLP基线模型（用于对比深度学习模型的提升效果）等诸多领域发光发热。

#### One-Hot独热编码

One-Hot编码又称为一位有效编码，在20世纪90年代左右被系统地论述和普及。这样的编码将一句话中各个特征的取值枚举出来，以二进制的方式进行编号。类似于定义了一份”座位表“，为各个特征的所有取值都确定好了对应的编号。这样在处理文本时，先把文本分词得到一系列特征词，然后逐个查询它们在”座位表“中的One-hot编码，从而得到对应的向量，拼在一起作为整个文本转换而来的向量：

```text
One-Hot编码向量化示例
---------------------
做出如下规定：
	性别：男性(10), 女性(01)
	国籍：中国(100), 美国(010), 法国(001)
	运动：足球(1000), 篮球(0100), 羽毛球(0010), 乒乓球(0001)
  
那么对于“一位男性中国乒乓球员”：
	系统会将其分词为["男性", "中国", "乒乓球"],
	从而得到向量[1, 0, 1, 0, 0, 0, 0, 0, 1]
```

需要说明的是，在传统的机器学习（如逻辑回归、决策树）中，类别特征是没有大小顺序的。如果把“足球”编码成1，“篮球”编码成2，“羽毛球”编码成3，算法就会误以为“足球 > 篮球 > 羽毛球”，即引入了错误的数值大小关系。而One-Hot通过强制让两两正交（向量夹角余弦=0），明确定义“足球”、“篮球”、“羽毛球”是三个完全独立、平级、不存在大小优劣之分的类别。这正是One-Hot编码最本质的设计初衷。

不足的是，在One-Hot编码中，每个词被表示为一个唯一固定的向量（类似于统一的词典），它的取值与上下文完全无关。这样的编码策略使得One-Hot无法解决一词多义问题。例如，“苹果”这个词在“吃苹果”和“苹果手机”中向量完全一样。与词袋模型的一个区别是，One-Hot只会记录各个分类的取值情况，不会记录词语出现的频率。

#### TF-IDF

词袋模型的问题源于只统计每个词语出现的频率，导致一些常用词的词频普遍过高，盖住了不同文本之间的差异，使得转化成的向量特征不明显。TF-IDF模型在词袋模型的基础上进行升级，它的核心思想是：在一个文档中出现多、但在所有文档中都少见的词（如"神经网络"）更重要；到处都出现的词（如"的"、"了"）即使大量出现，也不能作为文本的特征来处理。TF-IDF分为以下两部分：

- **TF**（Term Frequency，词频）：词语在文档中出现的频率（词语出现次数 / 文档总词语数）
- **IDF**（Inverse Document Frequency，逆文档频率）：log(总文档数 / 包含该词的文档数)

将TF和IDF相乘，就得到了一个词在特定文档中的TF-IDF值。这个值越高，表示这个词对于区分文档越重要。如此处理，通过聚焦更具有特征的词汇，TF-IDF的方案比纯词频分析提升了数倍效果，至今仍在信息检索中广泛使用。

TF-IDF如同词袋模型、One-Hot独热编码一样，在处理超长文本时需要为大量的特征词汇增加维度，导致分析得出的向量维度会非常大。

### ② 第二阶段：词嵌入（Word Embedding）—— 静态稠密表示

词袋模型只是对文章出现的词语按照频率和重要程度进行统计，实际上并没有做到“语义”上的分析。在这些方案中，一个类型不同的取值被认为是相同的关联度，并没有进行“分类”。例如，在处理“这本书很好”、“这本书很棒”以及“这本书很烂”三句话时，这些策略会认为“好、棒、烂”是同类词语中的不同词项，具有相同的关联度，只会觉得两句话**组成结构**相似,无法识别“好”与“棒”相似，而“烂”的意思完全相反，从而无法判别出前两句话**语义相似**，后一句话**语义相反**。

为了解决这个问题，Google公司在2013年提出了Word2vec算法。该算法的主要步骤如下：

1. 基于语料库（包含海量文本数据）构建词表（每个词语对应的向量）
2. 对词表中的向量进行随机初始化，得到每个单词随机生成的向量
3. 构建神经网络模型，核心功能是根据一段文本中的“周围词”，来预测“中间词”
4. 对模型进行训练，调整词表中的向量：
	1. 对于输入的两个词语，匹配它们在词表中的向量
	2. 将这两个向量作为周围词输入模型，模型输出预测的中间词
	3. 将预测的中间词与训练文本中的目标词对比，计算损失（前向传播），根据损失更新词表中的向量（反向传播）
5. 在经过大量的“输入-匹配-预测-损失-更新”训练后，词表中各个词语的向量取值会逐渐稳定（收敛），得到一个稳定的词表矩阵
6. 后续进行文本向量化任务时，直接使用词表矩阵中的向量进行匹配即可。

这里的神经网络模型分为两种，上例中根据“周围词”预测“中间词”称为“CBOW”，反之根据“中间词”预测“周围词”称为“Skip-gram”。

```text
Word2Vec处理过程示例
---------------------
设语料库中有如下内容：
	这本书很好
	这本书很棒
	这部电影很好
	这部电影很棒
	
训练过程中，模型发现“好”、“棒”经常出现在“很”的附近，会因为“好”、“棒”之间的损失降低，而将这两个词的向量逐渐调整得比较接近，最终得到的词向量如下：

好 → [0.81, 0.23, ...]  
棒 → [0.79, 0.25, ...]

在词袋模型中，这两个词的向可能会被设定为[0,1]和[1,0]，于是它们之间的夹角余弦值为0。然而，通过Word2Vec处理得到的词向量，cos(好,棒) ≈ 0.95。这个值很接近于1，表明Word2Vec不再依赖词频统计，而是通过“上下文共现关系”学习词语的语义表示，从而让语义相近的词在向量空间中彼此靠近。
```

但是需要注意的是，对于“这本书很差”这样的句子，我们发现“好、棒、坏、差”往往都在“很、非常、比较”这样的词附近，即处于极度相似的上下文中。Word2Vec 并不会天然知道“好”和“差”是反义词，甚至会觉得它们也是相似的。也就是说，Word2Vec学到的是“语义角色”，不是“情感极性”。它能感觉到“好、棒、坏、差”都是一些评价用的词语，而”很、非常、比较“这些词语都更偏向于用来描述程度。另外，

即便如此，Word2Vec与词袋模型或One-Hot编码相比仍然有很大的突破：假设有一个语料库中包含200个城市名称。采用One-Hot编码时，每个城市都需要使用200维向量表示，其中只有一个分量取值为1，其余分量均为0，因此任意两个城市词语之间的相似度是一致的（词向量夹角余弦值都为0），无法体现它们之间的语义关系。而Word2Vec可以将每个城市映射到固定长度的稠密向量（例如5维、100维或300维）。在训练过程中，如果“北京”“上海”等词语经常出现在“中国、首都、经济、长三角”等相似语境中，它们的向量会逐渐靠近；如果“纽约”“旧金山”等词语更多出现在“美国、硅谷、华尔街”等语境中，它们也会形成另一组相近的向量。因此，Word2Vec不仅缓解了One-Hot编码带来的维度爆炸问题，还能够利用上下文信息学习词语之间的潜在语义关系，使语义相近的词在向量空间中自然聚集。

进一步来说，相比于同级城市词向量之间的聚类效果，Word2Vec的厉害之处在于它能够理解关系向量：

```text
北京 - 中国  ≈  东京 - 日本  ≈  巴黎 - 法国

因为  国王 - 男性  ≈  王后 - 女性，
所以  国王 - 男性 + 女性  ≈  王后。
```

这说明向量空间中不仅保存了“谁和谁相似”，还保存了“谁和谁是什么关系”。这样”层次关系感知“的能力，是词袋模型/On-Hot编码完全无法实现的，因此当年的Word2Vec在NLP（自然语言处理）领域震动四方。这种基于分布假设的预测性表示，达到了词向量静态嵌入的巅峰。

提及第一代词嵌入模型，通常与Word2Vec被放在一起介绍的还有GloVe和FastText。它们的核心目标都是将词语映射为低维稠密向量，使语义相近的词在向量空间中彼此接近。Word2Vec通过上下文预测学习词向量，GloVe通过全局共现统计学习词向量，FastText则在Word2Vec基础上引入子词机制，使模型能够利用词内部结构并处理未登录词。三者共同推动了从One-Hot表示向语义向量表示的转变，为后续第二代词嵌入模型（ELMo、BERT等）和现代大模型Embedding（OpenAI Embedding、BGE等）奠定了基础。

### ③ 第三阶段：基于上下文（Context-Based）—— 动态表示

在前文提到的第一代词嵌入模型（Word2Vec等）中，由于每个词语在词表中只对应了一个向量，Word2Vec也没有很好地处理一词多义问题。于是第二代词嵌入模型应运而生。其中ELMo、GPT和BERT都是在2018年几乎同时登场的三个里程碑模型。

ELMo的核心贡献是解决了传统静态词向量（如Word2Vec）无法区分一词多义的难题。它使用双向LSTM（Long Short-Term Memory，长短期记忆神经网络），为每个词生成一个上下文相关的动态向量。例如，"苹果"在"吃苹果"和"苹果公司"中的向量表示是不同的。在应用时，ELMo更像一个"特征提取器"，为下游任务提供更优质的词嵌入特征。

GPT开创性地证明了单向Transformer在大规模预训练后的强大生成能力。它采用Transformer的解码器部分，在预训练时只允许每个词看到它左边的词，因此非常擅长根据上文续写或生成连贯的文本。这种架构为后来风靡全球的ChatGPT等生成式大模型奠定了基础。

BERT的设计理念与GPT针锋相对，它的目标是深度理解语言。它的核心创新是掩码语言模型（MLM）：随机遮盖输入中15%的词，让模型根据左右两侧的上下文去"填空"。这迫使模型建立真正的双向理解能力。同时，下一句预测（NSP）任务让它能理解句子间的关系。BERT在推出时，刷新了11项自然语言理解任务的纪录，成为了"理解"类任务的标杆。

### ④ 第四阶段：大模型Embedding —— 智能划分

如果说BERT时代的向量化技术是“让模型理解词义”，那么如今基于大模型的文本向量化技术，则是在追求让模型获得更强大、更通用、更精准的“文本语义表示”能力。

在过去，生成高质量的文本嵌入几乎被双向编码器（如BERT）垄断，因为它们能同时看见上下文。而现在，趋势已经转向了基于仅解码器（Decoder-Only）架构的大语言模型（如GPT、LLaMA、Mistral）。仅解码器模型本质上是“单向”的（只能看到上文），这看似与需要双向理解（同时参考上下文）的向量化任务相悖。为此，研究人员开发了多种技术来克服这个限制。例如，通过移除因果注意力掩码，让模型在训练嵌入时能够进行双向注意力计算；或者设计巧妙的提示词策略和池化方法，如prompt_eol（在提示末尾获取表示）和pcoteol（模拟思维链），来引导模型生成更好的嵌入。

## 4.长文本的处理：Chunk

注意到我们在探索embedding的时候，大多数例子都是基于一些简短的语句来展现的。那么对于一篇长文章，如何把它进行向量化呢？在文本向量化之前，通常需要引入Chunk（文本切块）技术，对长文本进行合理拆分，以提升后续向量检索的准确性和效率。

### ① Chunk的本质

Chunk技术本质上是将长文本按照一定规则，划分为多个相对独立且语义完整的文本片段（Chunk），然后分别对这些片段进行向量化处理。这样做的主要原因在于，Embedding模型通常存在输入长度限制，无法直接处理超长文档。同时，即便模型能够接受较长文本输入，将多个主题混合在同一个向量中也会导致语义表达被稀释，降低检索精度。例如，一份同时包含数据库设计、索引优化、数据备份和集群部署等内容的技术文档，如果整体生成一个向量，那么该向量实际上混合了多个主题的信息。当用户查询“联合索引优化”时，系统很难准确定位到与索引相关的具体内容。

### ② Chunk的实现方式

Chunk技术根据切分策略的不同可以分为多种实现方式：

- 最简单的是固定长度切分（Fixed Chunking），即按照预设的字数或Token数量进行切分，例如每500个Token生成一个Chunk。这种方法实现简单，但容易破坏原有语义结构，导致一句完整的话被拆分到两个Chunk中。
- 为了解决这一问题，实际应用中更常采用滑动窗口切分（Overlap Chunking）策略，即相邻Chunk之间保留一定比例的重叠内容。例如，当Chunk长度为800个Token时，可以设置100至200个Token的重叠区域，从而保证跨边界的重要信息不会丢失。
- 除了基于长度的切分方法外，还可以按照文档结构进行切分。例如，根据标题、章节、段落、表格、代码块或换行符等结构信息生成Chunk。这种方式能够较好地保持语义完整性，因此在技术文档、学术论文和规章制度等结构化文本处理中应用广泛。
- 近年来，随着Embedding技术的发展，语义切分（Semantic Chunking）逐渐成为研究热点。该方法通过计算相邻句子或段落之间的语义相似度，在检测到主题发生明显变化时自动进行切分，从而使每个Chunk尽可能保持主题一致。相比固定长度切分，语义切分通常能够获得更好的检索效果，但其计算复杂度和实现成本也相对较高。

值得一提的是，Chunk大小的选择是影响RAG系统性能的重要因素之一。如果Chunk过小，虽然检索粒度更细，但可能缺乏足够的上下文信息，导致内容表达不完整；如果Chunk过大，则可能包含多个主题，使Embedding向量难以准确表达核心语义。因此，实际应用中通常需要在上下文完整性与检索精度之间进行平衡。对于技术文档和知识库场景，Chunk长度一般控制在500至1000个Token之间，并配合10%至30%的重叠区域，以获得较好的检索效果。

### ③ Chunk在RAG中的地位

总体来看，Chunk技术是连接原始文档与Embedding模型之间的重要桥梁。Embedding负责将文本转换为向量空间中的语义表示，而Chunk则决定了哪些内容被组织在同一个语义单元中。从实际工程经验来看，合理的Chunk策略往往对RAG系统效果的提升作用不亚于Embedding模型本身。可以说，Embedding决定了语义表示的质量，而Chunk决定了语义表示的边界，两者共同构成了现代知识库检索系统的基础能力。

# 三、存储：向量数据库

对于一批文本数据，我们通常使用关系型数据库，将每一个文本作为一条记录存储；或者使用MongoDB这样专用的文档数据库存储。那么对于embedding得到的向量，如何方便地对它们进行存储和处理呢？

向量知识库是一种专门用于存储和检索高维向量数据的数据库系统。用于存储和处理由文本、图片、音频等非结构化的数据经过Embedding模型处理转换得到的一组数字向量。向量数据库库的核心任务包括存储这些向量、建立高效索引、根据向量相似度快速查找最相关的数据。因此，向量数据库库本质上是一个基于语义相似度检索的数据系统。常见的向量数据库包括Milvus、Pinecone、Weaviate、Qdrant以及Chroma。

向量数据库并非是用来取代关系型数据库（MySQL等）的，事实上在企业级AI应用中，这两者通常共同存在。用户信息、订单数据、商品信息等要求强一致性、事务支持、精确查询的数据，仍然适合使用关系型数据存储（不能把账号密码也转换成向量，然后登录的时候去检索一下相关账号吧）；产品说明书、操作手册、FAQ等文档，在需要进行语义理解、相似内容检索、自然语言搜索等操作时，使用专用的向量数据库就更加合适了。

|      | MySQL   | 向量知识库       |
| ---- | ------- | ----------- |
| 数据类型 | 结构化数据   | 向量数据        |
| 存储形式 | 行和列     | 高维向量        |
| 查询方式 | SQL     | 相似度搜索       |
| 检索逻辑 | 精确匹配    | 语义匹配        |
| 查询对象 | 数值、字符串  | Embedding向量 |
| 擅长场景 | 业务系统    | AI知识检索      |
| 返回结果 | 条件满足的数据 | 语义最接近的数据    |
| 数据关系 | 主键、外键   | 向量距离        |

关系型数据库解决的是“数据如何准确存储和查询”的问题，而向量知识库解决的是“数据之间语义上是否相似”的问题。前者依赖SQL和精确匹配，后者依赖Embedding和向量相似度计算；在RAG系统中，向量知识库承担着“语义检索引擎”的角色，是连接企业知识与大语言模型的重要桥梁。

# 四、检索：在知识库中进行向量相似度匹配

了解了RAG系统是如何把长文本切块，并且转换成向量存储之后，如何查找与一个向量相似的一些向量，是我们必须面临和解决的问题。向量相似度检索（Vector Similarity Search）是向量数据库的核心技术，其目标是在海量高维向量中快速找到与查询向量最相似的若干个向量（Top-K）。

在向量知识库和Embedding技术中，最常用的相似度计算方法之一就是余弦相似度（Cosine Similarity）。由于embedding的策略是对相似的文本进行”聚类“，相关的内容更容易被分配到相似的方向上。正因为这样，余弦相似度算法用于衡量两个向量在方向上的相似程度，而不比较它们的绝对大小。也就是说，余弦相似度关注的是“两个向量是否指向同一个方向”，而不是“两个向量长度相差多少”。通过下面的例子可以说明这一点：

```text
例：对于这两句话：
A:"人工智能"
B:"人工智能人工智能"

向量化的一种情况是：
A = [1, 2]
B = [2, 4]
```

从数量上看，`B = 2 * A`；但是A、B的语义相同性更多展现在它们的向量角度差异之上，而不是与长度差异有关。因此我们只比较方向、不需要受长度影响。

通过向量夹角的计算公式
$$
cos(A,B)=cos \theta= \frac{A \cdot B} {|A| \times |B|}​
$$
可知，两个向量之间会形成一个夹角θ越小，方向越接近；反之夹角θ越大，方向越不同。余弦相似度的核心算法就是计算cosθ。根据夹角θ与夹角余弦值cosθ的取值，可以总结出这样的对照关系：

| 夹角   | 余弦值 | 含义   |
| ---- | --- | ---- |
| 0°   | 1   | 完全相同 |
| 90°  | 0   | 无关   |
| 180° | -1  | 完全相反 |
于是要判断知识库中的两段文本之间是否相关，就可以转化为判断两个文本块embedding得到的向量是否具有一个比较小的夹角的问题，从而通过数学工具得以解决。

# 五、实战：RAG在实际项目中的落地

行文至此，通用RAG流程中所需要时用到的一些技术原理都在前文有或多或少的介绍。从文本到知识库检索，通常遵循这样的工作流程：

```text
原始文本文档
↓  
Chunk切块
↓  
按块进行Embedding（向量化）  
↓  
存入向量知识库  
↓
用户提问
↓  
将问题进行Embedding 
↓  
将问题向量与知识库中的海量向量进行余弦相似度匹配
↓  
找到最相似的K个（Top-k）Chunk文本块
↓  
将这些文本块作为相关知识，嵌入系统提示词中，
和用户问题一起送入大模型
↓    
大模型收到提示词，生成对应的回答
```

### ② 文章分块（Chunk）与向量化（Embedding）的核心逻辑

```python
def extract_article_data(self):
        """提取所有已发布文章数据"""
        try:
	        # 导入后端原有的Article模型 
            from .models import Article
            # 通过Article模型从数据库提取所有文章，按发布时间倒排（先新后旧）
            articles = Article.objects.filter(
                publish_date__lte=timezone.now()
            ).order_by('-publish_date')
            logger.info(f"提取到 {articles.count()} 篇已发布文章")
            return articles
        except Exception as e:
            logger.error(f"提取文章数据失败: {str(e)}")
            raise

def prepare_article_text(self, article) -> str:
	"""拼接文章文本：标题 + 摘要 + 内容"""
	parts = [article.title]

	if article.summary:
		parts.append(article.summary)

	parts.append(article.content)

	return "\n".join(parts)

def build_metadata(self, article, chunk_index: int, total_chunks: int) -> Dict[str, Any]:
	"""构建文本块元数据"""
	metadata = {
		"article_id": article.id,                         # 文章id
		"title": article.title,                           # 文章标题
		"category_id": article.category_id,               # 分类id
		"tags": article.tags or "",                       # 标签列表
		"publish_date": article.publish_date.isoformat(), # 发布时间
		"chunk_index": chunk_index,                       # 切块编号
		"total_chunks": total_chunks,                     # 文章总分块数
		"summary": article.summary or ""                  # 文章总结
	}

	# 尝试获取分类名称
	try:
		category_name = article.get_category_name()
		metadata["category_name"] = category_name
	except Exception:
		metadata["category_name"] = "未知分类"

	return metadata

def vectorize_article(self, article) -> int:
	"""对单篇文章进行向量化"""
	try:
		logger.info(f"开始向量化文章: {article.title}")

		# 准备文本
		article_text = self.prepare_article_text(article)

		# 文本分块
		chunks = self.text_splitter.split_text(article_text)
		logger.info(f"文章 '{article.title}' 分割为 {len(chunks)} 个文本块")

		# 构建文档列表
		documents = []
		for index, chunk in enumerate(chunks):
			metadata = self.build_metadata(article, index, len(chunks))
			documents.append({
				"page_content": chunk,
				"metadata": metadata
			})

		# 添加到向量数据库
		texts = [doc["page_content"] for doc in documents]
		metadatas = [doc["metadata"] for doc in documents]

		# add_texts()方法由Chroma自动管理，配置好embedding模型后，文本块数据入库时，Chroma会自动调用embedding模型进行向量化处理
		self.chroma_client.add_texts(
			texts=texts,
			metadatas=metadatas
		)

		# 更新文章向量状态
		article.is_vectorized = True
		article.vectorized_at = timezone.now()
		article.chunk_count = len(chunks)
		article.save(update_fields=['is_vectorized', 'vectorized_at', 'chunk_count'])

		logger.info(f"文章 '{article.title}' 向量化完成，生成 {len(chunks)} 个文本块")
		return len(chunks)

	except Exception as e:
		logger.error(f"文章 '{article.title}' 向量化失败: {str(e)}")
		raise
```

我们来看`build_metadata()`方法的功能：为chunk后得到的文本块构建**元数据**（meta data）。它是“描述数据的数据”，本身不是主要内容，而是用来说明、管理、组织和理解数据的信息。使用元数据来管理文本块有这样的好处：

1. 如果原始文档包含同样主题的不同版本，可以通过元数据标识出来各个文本块的版本和更新时间，后续召回时优先参考最新的版本，有利于旧版文档中的陈旧知识对生成回答造成干扰。
2. 现代RAG架构通常采取“关键词检索(BM25) +向量检索(Vector Search)”的双重知识召回策略，通过元数据进行关键词检索，有利于快速锁定时间、编号等需要严格匹配的关键词，从而找到相关的知识。
3. 企业级RAG几乎都会做权限分隔，不允许为普通用户展示需要保密的内容。通过配置元数据中的权限等级，在召回时根据用户的权限进行限制，只召回用户可看范围内的知识，避免出现“普通职员可以看内部财务报表”的问题。
4. 文档切分为文本块之后，会导致上下文信息丢失。我们无法得知一个文本块对应的向量来自于哪篇文章的第几个章节、它前面和后面的文本块是哪一个。使用元数据进行标记，就可以很方便地解决这个问题。

于是，向量数据库通过存储`向量+元数据+文本块`,不仅保留了文本块与向量之间的对应关系，也很好地通过元数据保留了文本块之间的联系，AI问答系统在知识库中检索和召回相关知识提供了很大的帮助。

## 3.向量知识库可视化管理

### ① 向量数据的可视化展示

文本块embedding之后得到的向量维度通常达到成百上千的数量级，如何可视化地体现出知识库中具有的大量知识，直观地看出所有的知识向量在空间上的分布情况，需要找到一种算法来实现向量知识库的可视化效果。在这里，我们考虑将知识库中的向量压缩到三维，前端使用ECharts 3D散点图展示。

直接将向量的维度分成三组，将每组分量线性组合，得到3个数字作为向量压缩后的3D坐标。这样的方法实现起来很简单，但是会破坏向量之间原有的相似度聚类关系，即打破了embedding模型构建出的相似度特征，这样放到空间中展示时，就不能认为相近的点语义仍然相似了。UMAP降维压缩算法就是此类问题的解决方案之一。

>UMAP（Uniform Manifold Approximation and Projection，统一流形近似与投影）是一种非线性降维算法，主要用于将高维数据压缩到低维空间（通常是2维或3维），同时尽可能保留数据的全局结构和局部邻域关系，以便进行可视化、聚类分析和特征压缩。

对于高维向量集合，UMAP算法首先计算每个样本最近的邻居，利用KNN（K近邻）算法寻找每个点只距离最近的一些邻居（邻居数量由n_neighbors参数的取值决定）。



### ② 向量知识库的管理

博客系统的文章经常会有更新。为了方便地将修改后的文章和新文章同步到向量知识库中，我在管理端添加了文章的向量化管理功能。点击按钮即可按需处理新上传的文章，进行增量更新；或者进行全量更新，重新向量化所有文章。
```python
def vectorize_all_articles(self, incremental: bool = False) -> Dict[str, int]:
	"""批量向量化所有文章"""
	try:
		articles = self.extract_article_data()

		if incremental:
			# 只处理未向量化或更新后的文章
			articles = articles.filter(
				Q(is_vectorized=False) |
				Q(updated_at__gt=F('vectorized_at'))
			)
			logger.info(f"增量模式：处理 {articles.count()} 篇文章")
		else:
			# 全量模式：处理所有文章
			logger.info(f"全量模式：处理 {articles.count()} 篇文章")

		total_articles = articles.count()
		total_chunks = 0
		success_count = 0
		failed_count = 0

		for article in articles:
			try:
				chunks_count = self.vectorize_article(article)
				total_chunks += chunks_count
				success_count += 1
			except Exception as e:
				logger.error(f"文章 ID={article.id} 向量化失败: {str(e)}")
				failed_count += 1

		result = {
			"total_articles": total_articles,
			"success_articles": success_count,
			"failed_articles": failed_count,
			"total_chunks": total_chunks
		}

		logger.info(f"向量化完成: {result}")
		return result

	except Exception as e:
		logger.error(f"批量向量化失败: {str(e)}")
		raise
```

def search_similar_articles(self, query: str, top_k: int = 5, filters: Optional[Dict] = None):
	"""语义搜索文章"""
	try:
		logger.info(f"执行语义搜索: query='{query}', top_k={top_k}")

		if filters:
			# 使用过滤条件进行搜索
			results = self.chroma_client.similarity_search_with_score(
				query=query,
				k=top_k,
				filter=filters
			)
		else:
			# 无过滤条件的搜索
			results = self.chroma_client.similarity_search_with_score(
				query=query,
				k=top_k
			)

		logger.info(f"语义搜索完成，返回 {len(results)} 个结果")
		return results

	except Exception as e:
		logger.error(f"语义搜索失败: {str(e)}")
		raise

def clear_vector_database(self):
	"""清空向量数据库"""
	try:
		logger.warning("清空向量数据库")
		# 删除现有集合
		self.chroma_client.delete_collection()

		# 重新创建集合（使用与初始化时相同的导入方式）
		try:
			from langchain_chroma import Chroma
		except ImportError:
			from langchain_community.vectorstores import Chroma

		self.chroma_client = Chroma(
			persist_directory=Config.CHROMA_PERSIST_DIR,
			embedding_function=self.embeddings,
			collection_name="articles"
		)

		# 重置所有文章的向量化状态
		self.ArticleModel.objects.all().update(
			is_vectorized=False,
			vectorized_at=None,
			chunk_count=0
		)

		logger.info("向量数据库已清空")

	except Exception as e:
		logger.error(f"清空向量数据库失败: {str(e)}")
		raise

def get_vectorization_status(self) -> Dict[str, Any]:
	"""获取向量化状态统计"""
	try:
		from .models import Article

		total_articles = Article.objects.count()
		vectorized_articles = Article.objects.filter(is_vectorized=True).count()
		pending_articles = total_articles - vectorized_articles

		# 获取最近一次向量化时间
		last_vectorized = Article.objects.filter(
			is_vectorized=True
		).order_by('-vectorized_at').first()

		last_update = last_vectorized.vectorized_at if last_vectorized else None

		# 统计总文本块数
		total_chunks = Article.objects.filter(
			is_vectorized=True
		).aggregate(total=Sum('chunk_count'))['total'] or 0

		return {
			"total_articles": total_articles,
			"vectorized_articles": vectorized_articles,
			"pending_articles": pending_articles,
			"total_chunks": total_chunks,
			"last_update": last_update.isoformat() if last_update else None,
			"storage_directory": Config.CHROMA_PERSIST_DIR,
			"embedding_model": Config.EMBEDDING_MODEL_NAME
		}

	except Exception as e:
		logger.error(f"获取向量化状态失败: {str(e)}")
		raise
