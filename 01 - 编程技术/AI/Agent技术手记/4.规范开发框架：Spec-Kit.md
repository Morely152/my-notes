在使用AI智能体进行开发时，经常会出现这样的情景：

1. 只对AI说“添加一个XX页面”，写出来的风格与网站不搭，功能也不如人意；
2. 组件化效果不好，光“加载动画”就有四五种，组件主题不统一；
3. 后端接口路径参差不齐，权限控制混乱，需要反复修改规整。

如同人类程序员进行项目开发时遵循的“需求 - 编码 - 测试 - 提交”一样，AI也需要一套明确的规则来进行约束和指导，这样我们的AI助手才能化身为专业的程序员，从而准确高效地完成任务。由Github推出的规范驱动开发框架Spec-kit，以及后来更加轻量化的OpenSpec，就起到了这样的作用。

## 1.初始化项目

```bash
uvx --from git+https://github.com/github/spec-kit.git specify init [项目名称]
```

## 2.项目结构解析

以下为新建的spec-kit项目文件结构，以及对应的文件作用：

```text
.claude/                                     # Claude助手工作流配置目录，存放各类命令指导文档
 ├─ commands/                                # 命令说明文档目录，定义OpenSpec各阶段操作指南
 │   ├─ speckit.analyze.md                   # 指导分析现有系统/代码库，识别组件与问题
 │   ├─ speckit.checklist.md                 # 规范制定/实施的检查清单，保障工作完整性
 │   ├─ speckit.clarify.md                   # 澄清需求/规范中的模糊点，统一各方认知
 │   ├─ speckit.constitution.md              # 定义项目核心原则、决策流程等治理框架
 │   ├─ speckit.implement.md                 # 指导将规范转化为具体实现步骤，落地规范
 │   ├─ speckit.plan.md                      # 制定实施计划，包含任务分解、时间线、里程碑
 │   ├─ speckit.specify.md                   # 指导编写清晰可验证的技术规范（核心文档）
 │   ├─ speckit.tasks.md                     # 定义管理项目具体任务，含描述、负责人、截止日期
 └   └─ speckit.taskstoissues.md             # 指导将任务转化为可跟踪的Issue，便于协作管理
 
 .specify/                                   # OpenSpec核心目录，存放规范制定相关文件/模板
 ├─ memory/                                  # 项目记忆目录，存储上下文、决策记录等参考资料
 │   └─ constitution.md                      # 项目宪法具体内容，定义治理规则与协作模式
 ├─ scripts/                                 # 自动化脚本目录，辅助OpenSpec工作流执行
 │   └─ powershell/                          # PowerShell自动化脚本目录
 │       ├─ check-prerequisites.ps1          # 检查OpenSpec运行的前置条件/依赖是否满足
 │       ├─ common.ps1                       # 通用函数/变量模块，供其他脚本复用
 │       ├─ create-new-feature.ps1           # 自动生成新功能规范文件、任务列表等
 │       ├─ setup-plan.ps1                   # 初始化项目实施计划，生成计划文件与任务分解
 │       └─ update-agent-context.ps1         # 更新AI助手上下文，确保信息最新
 ├─ templates/                               # 文档模板目录，保障规范/计划的结构一致性
 │   ├─ agent-file-template.md               # AI助手配置文件模板，定义角色/能力/上下文
 │   ├─ checklist-template.md                # 检查清单模板，用于创建自定义质量验证清单
 │   ├─ constitution-template.md             # 项目宪法模板，提供治理框架标准结构
 │   ├─ plan-template.md                     # 实施计划模板，定义任务分解/时间线标准格式
 │   ├─ spec-template.md                     # 技术规范核心模板，含目标/范围/需求/接口等
 └   └─ tasks-template.md                    # 任务列表模板，定义任务管理的标准格式
```

总结
1. .claude/commands/ 是 OpenSpec 各阶段操作的指导手册，覆盖从需求澄清到任务落地的全流程；
2. .specify/templates/ 是核心模板目录，保障产出的规范 / 计划文档有统一结构，是 OpenSpec 的基础骨架；
3. .specify/scripts/ 是自动化辅助脚本，能减少重复操作，提升 OpenSpec 工作流的执行效率。

## 3.标准化的开发流程

完成项目初始化之后，我们可以使用这个模板来进行标准化开发，主要有以下七个步骤。

| 步骤    | 环节         | 角色/视角     | 核心目的                                            | 关键产出                              | 实践建议                                          |
| ----- | ---------- | --------- | ----------------------------------------------- | --------------------------------- | --------------------------------------------- |
| **1** | **制定宪法**   | 项目创始人     | 为项目建立最高准则，奠定所有技术决策的基石。                          | `.specify/memory/constitution.md` | 将团队技术栈偏好、部署要求、第三方库选用原则等写入宪法，约束AI的技术方案。        |
| **2** | **创建功能规范** | 产品经理      | 用自然语言清晰定义“做什么”，描述功能、用户故事和验收标准，不涉及技术实现。          | `specs/[功能名]/spec.md`             | 描述越具体越好，例如“按钮点击后，任务项变灰并移至底部”，而非笼统的“用户可以完成任务”。 |
| **3** | **澄清需求**   | 测试人员/产品经理 | （可选）消除 spec 中的模糊地带和歧义，通过提问来完善需求。                | 对 `spec.md` 的补充和修正。               | 强制自己在编码前想清楚所有边界情况和异常流程，有效减少后期返工。              |
| **4** | **生成技术方案** | 架构师       | 基于宪法和规范，设计“怎么做”，包括技术选型、架构设计、数据模型、API契约等。        | `plan.md`、`data-model.md` 等设计文档。  | 如果AI的方案（如库的选择）不符合预期，可以直接要求修改，你拥有最终决策权。        |
| **5** | **分析计划**   | 审查者       | （可选）在分派任务前进行“沙盘推演”，检查所有工件（spec, plan）是否存在矛盾或遗漏。 | 一份指出潜在问题的分析报告。                    | 强烈建议不要跳过。它像代码审查一样，能在早期发现许多高成本的错误。             |
| **6** | **生成任务列表** | 项目经理      | 将技术方案分解为详细、可执行、有依赖关系的具体施工步骤。                    | `tasks.md`                        | `tasks.md` 是与AI协作的核心界面。你可以随时介入，手动调整优先级或标记任务。  |
| **7** | **执行实现**   | 程序员       | 根据任务列表，按顺序、有条不紊地执行开发任务，生成最终代码。                  | 可运行的应用程序代码。                       | 对于复杂任务，建议让AI一个一个地执行，以便随时检查每一步的产出，确保过程可控。      |

使用以上七步，通过Spec-kit开发待办清单网页的流程示例如下[^3]：
### ① `/speckit.constitution`：制定项目标准

```bash

```

```bash
- 创建任务（包含标题与描述）
- 删除任务
- 查看任务

- 标记任务为完成
```

## 4.Spec-Kit总结：

### ① Spec-Kit发挥的作用

> Spec-Kit不是一个新的AI编程工具，它是一套工作流和方法论。[^4]

在这之前，我使用AI来进行开发时，都是先与Gemini、DeepSeek这样的大模型围绕项目交流一番，然后再让大模型写出项目的需求分析、结构设计等文档，这样做一定程度上可以让AI写的东西更符合我们的预期，但是不如现在的Spec-Kit完善规范。“（Spec-Kit）是一种工程思维的转变”[^3]。

通过**先规划再执行**的策略，我们先引导AI思考：
1. 要注意什么 -> 项目标准（constitution）
2. 要做什么内容 -> 项目功能（specify）
3. 用什么做 -> 技术方案（plan）
4. 怎么去做 -> 任务列表（tasks）

![](20260223110652480.png)

于是，AI就可以在这些文档的帮助下，按照我们制定的规范和提出的需求来编写项目代码，并且进行测试和验收，项目的稳定性和统一性要比跟AI边说边做的“凭感觉编程”好一些。

但是，使用Spec-Kit开发简单应用，花费的时间比"边说边做"又要长了。如果在进行大型项目开发、团队开发，或者追求代码质量和项目的规范性，适合使用SPec-Kit；而在进行"**Bug修复、UI微调、简单项目开发、需求不明确的探索式开发**"时，或许使用传统的跟AI“边聊边做”会更方便一些。

#### ② “宪法”的重要性 —— Constitution

现在回头看我们一开始写的Constitution：

```bash
/speckit.constitution 禁止使用蓝紫色渐变风格UI，制定以代码质量、测试标准、用户体验一致性及性能要求为核心的原则，整个网站使用中文
```

实际上这样写很草率，没有完全发挥出Constitution的“约束性”和“指导性”。当我们写Constitution时,我们要考虑的不是“我要写什么”，而应该是“**当 AI、工具和我意见不一致时，到底该听谁的？**”[^5]。这个“宪法”解决的不是"怎么写更好"的实现问题，而是"这样做行不行"的原则问题[^5]。

由此可见，Constitution 的核心角色有两个：

- 基石：为项目定义一套所有参与者（包括 AI）都必须遵守的、不可动摇的质量和原则底线。
- 裁决者：当出现模棱两可或冲突的情况时，为 AI 提供最高决策依据，告诉它应该"站哪一边"。

一份好的 `Constitution` 应该只关心三件事：

1. **价值排序（Priorities）**
    - 类型安全 > 开发速度
    - 可维护性 > 一时便捷
    - 系统一致性 > 个人风格
2. **不可谈判的边界（Non-Negotiables）**
    - 不能绕过类型系统。
    - 不能引入平行的技术体系（如在 React 项目中引入 Vue）。
    - 不可测试的代码视为不完整。
3. **冲突时的取向（Defaults）**
    - 当工具默认语言与团队主语言冲突时，优先使用团队主语言（如中文）。
    - 当可配置性与硬编码有冲突时，优先选择可配置性。

这里转载一下@方始终_大佬写的一份Constitution，原文链接详见“参考资料”部分:

```markdown
# Universal AI-Driven Engineering Constitution  
**Constitution v1.3（Tool-Agnostic · Project-Agnostic）**

> **权威声明（Source of Truth）**  
> 本文件定义 AI 与人类在软件工程中的协作治理原则，  
> 是所有基于 spec-kit 或等价规范驱动流程的项目的  
> **最高裁决协议（Single Source of Truth）**。  
>  
> 当效率、工具默认行为、个人偏好与本宪法发生冲突时，  
> **以本宪法为最高优先级**。

---

## I. 类型优先，安全第一  
**Type-First, Safety-First（NON-NEGOTIABLE）**

### 裁决原则
- 类型安全 **优先于** 开发速度  
- 编译期确定性 **优先于** 运行时容错  
- 明确约束 **优先于** 模糊灵活  

### 不可违背的边界
- 所有可类型化系统必须启用严格类型检查  
- 不允许绕过类型系统规避问题（如 `any`、隐式弱类型）  
- 所有对外边界（API、模块、组件、状态）必须具备明确契约  

> **裁决说明**：  
> 当“先跑起来”和“类型完整性”发生冲突时，必须选择后者。

---

## II. 配置驱动，组合优先  
**Configuration-Driven, Composition-First（NON-NEGOTIABLE）**

### 裁决原则
- 组合优于继承  
- 声明式优于命令式  
- 配置优于硬编码  

### 不可违背的边界
- 业务能力必须通过组合式抽象构建  
- 系统行为应可通过配置表达  
- 禁止将核心业务规则硬编码进表现层或入口逻辑  

> **裁决说明**：  
> 当“快速写死逻辑”和“可组合、可配置”发生冲突时，必须选择后者。

---

## III. 语义清晰，体系一致  
**Semantic Clarity, System Consistency（NON-NEGOTIABLE）**

### 裁决原则
- 可读性优先于技巧性  
- 一致性优先于个人风格  
- 语义优先于实现细节  

### 不可违背的边界
- 命名必须表达业务或领域语义  
- 局部实现不得破坏整体体系一致性  
- 禁止引入“只有作者能理解”的隐式约定  

> **裁决说明**：  
> 当个人偏好与系统一致性发生冲突时，以系统一致性为准。

---

## IV. 分层防御，用户友好  
**Layered Defense, User-Friendly（NON-NEGOTIABLE）**

### 裁决原则
- 系统稳定性优先于功能完整性  
- 用户可理解性优先于技术精确性  

### 不可违背的边界
- 错误必须被分层捕获与处理  
- 系统不得以“静默失败”作为默认策略  
- 单点错误不得导致系统级崩溃  

> **裁决说明**：  
> 当实现复杂度与系统韧性发生冲突时，必须选择系统韧性。

---

## V. 单一数据源  
**Single Source of Truth（NON-NEGOTIABLE）**

### 裁决原则
- 状态集中管理  
- 数据来源唯一  
- 派生而非复制  

### 不可违背的边界
- 不允许多个权威来源描述同一事实  
- 表现层不得成为事实裁决者  
- 所有状态变化必须可追踪、可审计  

---

## VI. 关注点分离  
**Separation of Concerns（NON-NEGOTIABLE）**

### 裁决原则
- 每一层只解决一个问题  
- 层与层通过明确契约协作  

### 不可违背的边界
- 领域逻辑不得渗透进展示或交互层  
- 基础设施、业务规则、表现逻辑不得混杂  

> **裁决说明**：  
> 当“减少抽象层级”和“职责清晰”发生冲突时，必须选择职责清晰。

---

## VII. 可测试性优先  
**Testability-First（NON-NEGOTIABLE）**

### 裁决原则
- 可验证性是设计质量的一部分  
- 不可测试的实现视为不完整  

### 不可违背的边界
- 核心业务逻辑必须可被自动化验证  
- 关键用户路径必须具备端到端验证能力  

> **裁决说明**：  
> 当设计导致难以测试时，必须重构设计，而不是削弱测试。

---

## VIII. 技术裁决权  
**Technology Decisions（NON-NEGOTIABLE）**

### 裁决原则
- 技术选择必须服务于长期可维护性  
- 不得引入平行技术体系制造分裂  

### 不可违背的边界
- 项目级技术裁决必须显式声明  
- 未经治理流程，不得随意替换核心技术路线  

> **说明**：  
> 具体技术栈由各项目在其自身规范中声明，但必须服从本宪法。

---

## IX. 文档语言裁决  
**Documentation Language Policy（NON-NEGOTIABLE）**

### 裁决原则
- 清晰理解 **优先于** 工具默认行为  
- 团队共识 **优先于** 外部惯例  

### 不可违背的边界
- 团队内部规范性文档默认使用 **团队主语言**  
- 关键技术术语可保留原语言，但必须提供语义说明  

### 例外说明
- 对外文档或跨语言协作场景  
  可根据受众选择单语或双语形式  

> **裁决说明**：  
> 当 AI 工具未明确指定输出语言时，  
> **以团队主语言作为最终交付语言**。

---

## X. 治理与修订  
**Governance（NON-NEGOTIABLE）**

### 宪法地位
- 本宪法为最高治理协议  
- 优先级高于所有 Rules、Guidelines 与工具约定  

### 修订流程
1. 提议：说明动机与影响范围  
2. 讨论：评估对系统一致性与长期成本的影响  
3. 审批：由治理责任人裁决  
4. 实施：更新宪法与相关配套规范  
5. 迁移：必要时制定迁移方案  

### 版本策略
- **MAJOR**：治理原则或裁决权变更  
- **MINOR**：新增或强化原则  
- **PATCH**：澄清、措辞与结构优化

## XI. 执行性说明  
**Execution Note**

- 本宪法定义 **“应当如何裁决”**，而非 **“具体如何实现”**  
- 具体实现规范（Rules）：  
  - 由当前工具或环境加载  
  - 其物理位置、格式与结构不属于宪法治理范围  
- 所有 AI / Agent：  
  - 必须以本宪法作为最高约束  
  - 不得自动生成、修改或绕过宪法内容
```
