事情是这样的，一直看网上都在吹OpenClaw很厉害，正好手上有一个闲着的服务器，看了一下部署也挺简单的，装一个npm包填一下API key就可以了。结果刚把网页端配置好之后就报错了，花了一下午+一晚上的时间才解决，最后查出来的原因真是给我气笑了😂。以下是“事故现场还原”：

## 顺利的开始…

刚开始的时候，安装Node.js、npm安装openclaw、配置API Key……一切行云流水，很快就给出了网页的url。配置一下站点和Nginx，系统的安装顺利完成，网页端准备就绪。

## 然后就开始不对劲了

打开浏览器，输入OpenClaw给出的网页端地址，页面加载出来了——然鹅这个状态好像不太对？

![](20260218210725784.png)

看到`no reason`的时候心里就咯噔一下，感觉这个问题会比较麻烦。

众所周知，报错信息越详细，事儿就越小。这回给个“没道理”该咋解决？

赶紧用F12打开开发者工具，控制台一片红：

![](20260218213403124.png)

坏了，这个WebSocket连接报错可咋整…
## 第一轮排查：OpenClaw的配置/Token验证

感觉像是访问网页端的时候没有加OpenClaw的Token：

```
之前的url：
https://域名:端口

改成：
https://域名:端口/?token=…
```

`shift+F5`强制刷新页面，不起作用。怀疑是服务器的OpenClaw配置有问题，于是打开终端看了一下：

```bash
systemctl status openclaw     # 服务运行中 ✓
netstat -tlnp | grep 18789    # 127.0.0.1:18789监听中 ✓
curl http://127.0.0.1:18789/health  # 返回健康状态 ✓

npm install -g wscat
wscat -c "ws://127.0.0.1:18789/?token=xxx"  # Connected！✓
```

好家伙，服务器本地WebSocket连接完全正常。那问题应该不在OpenClaw。

## 第二轮排查：Nginx反向代理

看到浏览器地址栏前面的https，感觉有点可疑，换成http之后虽然也报错了，但是报错信息更加清晰，明确告诉需要使用https或者本地连接：

![](20260218201432792.png)

问了一下AI，跟我说OpenClaw自己不支持HTTPS，但又强制要求HTTPS环境（这你扯不扯）。所以必须用Nginx的反向代理来提供HTTPS。

检查Nginx配置，WebSocket必需的几行都加上了：

``` nginx
proxy_http_version 1.1;
proxy_set_header Upgrade $http_upgrade;
proxy_set_header Connection "upgrade";
```

语法检查`nginx -t`通过，重载配置，甚至担心不够彻底，干脆重启了服务器……然而问题依旧，还是1006 no reason “没道理”。

## 第三轮排查：奇了怪了？

### 陷入僵局

Nginx配置没问题，OpenClaw服务正常，防火墙放行了，安全组配了，Token没错，甚至故意换了个错误的Token也一样报错`no reason`……所有技术层面的排查都做完了，该试的都试了，问题纹丝不动。

这时候已经过去四五个小时，心态开始有点崩。

### 转折点：来自社区的一条评论

寻思AI还是解决不了这种冷门问题，于是去翻帖子，看到这么一句：

> "请务必安装Google Chrome的官方.deb包，而不是使用apt install chromium或Snap包。因为Snap版的Chromium有严格的沙盒限制，会导致OpenClaw无法正常连接和控制" [^1]

![](20260218210423064.png)

突然想起来，之前无意中使用Edge打开OpenClaw的网页时报了另一个错误……我现在用的Chrome好像不太对劲！之前重装系统后为了省事，Chrome是从一个**第三方下载站**下的安装包，不是Google官方渠道。当时一看能用就没管它。

### 验证时刻

打开电脑上自带的Microsoft Edge，页面加载……状态变成了绿色的`正常`。
再掏出手机，用后来下载的官方Chrome打开——也成功了！
用回那个第三方Chrome——还是1006。

破案了。

![](20260218211443051.png)
### 真相大白

折腾了一下午+一晚上的问题，最后的根源是：**浏览器有问题**。

不是假的Chrome，但可能经过了第三方的魔改或者移植，WebSocket实现有缺陷，或者沙盒限制太严，导致OpenClaw的WebSocket握手永远无法完成。

真正的解决方案：**换个正经的浏览器**（不是说Chrome不正经，是我不知道在哪下的魔改Chrome的锅…）。

### 一些感慨

1. **最复杂的问题往往有最简单的答案**。几小时的深度技术排查，最后被一个“换浏览器”解决。
2. **尽量使用官方的原厂APP**。尤其是从非官方渠道下载的软件，鬼知道被改了什么。
3. **多浏览器测试应该成为排查的标准流程**。这次如果用Edge或Firefox早试一下，起码能省下80%的时间。
4. **社区经验很重要**。最后那条救命评论来自一个不起眼的帖子，如果不是翻到它，可能现在还在跟Nginx配置死磕。
5. **不要过于依赖AI**。AI的大多知识源于网络，对实际的计算机环境没有我们自己清楚，对于这样“偏门”的问题，也要记得找老方法查各大社区的文章和帖子，不要根AI死磕一下午。

如果你也在部署OpenClaw（或者其他依赖WebSocket的自托管应用）时遇到1006错误，服务器端排查无果，不妨先试试：

- 换个浏览器（Edge、Firefox、手机版Chrome都行）    
- 确保用的是官方渠道下载的Chrome（.deb包或官网直接下载）
- 浏览器控制台里跑一下WebSocket诊断代码：

说不定能省下一整个下午。
